# Phase 5 Context — Multiple-Choice Interview UX

## Problem Statement

The current interview flow presents each question with a blank textarea, requiring the interviewer to type the elderly person's full response. This creates high friction:
- Typing is slow, especially for interviewers who aren't fast typists
- Open-ended boxes feel intimidating ("what do I write?")
- The elderly person may not know how to articulate preferences without prompts
- Each question feels like a mini-essay assignment

## Solution: AI-Generated Multiple Choice + "其他" Escape Hatch

Each question will come with 3-4 predefined answer options generated by the AI based on the domain, question context, and person's background. The interviewer reads the options to the elderly person, who picks one. If none fit, "其他" reveals a text input.

## Design Decisions

### D-01: Options are generated server-side alongside questions
The `/api/interview/questions` route already generates 4 questions per domain. We extend the response format so each question includes an `options` array of 3-4 strings. The AI prompt is updated to generate contextually appropriate options.

### D-02: Options are answer suggestions, not constraints
Options represent common/reasonable answers for the question. They should feel like "most people in this situation might say..." rather than a rigid multiple-choice exam. The "其他" option ensures no one is forced into a box.

### D-03: Selected option text becomes the answer
When the user selects an option, its text is written to the answer field (same `updateAnswer()` store call). The transcript format is unchanged — downstream archive generation and query work without modification.

### D-04: "其他" reveals a textarea
Clicking "其他" shows the existing textarea component. The user can type freely. This is the current behavior, just gated behind one extra click.

### D-05: Options layout is vertical card/chip list
Each option is a tappable card with the full text. Selected state uses sky-500 border + light sky background. This is more touch-friendly than radio buttons for potentially long option text.

### D-06: No changes to followup probes
The ProbePanel (follow-up questions) remains free-text. Probes are short and contextual — multiple choice doesn't make sense there.

### D-07: Backward compatible transcript
The `Turn` type and `SessionData.transcript` structure are unchanged. Whether the answer came from a selected option or custom text is not distinguished in the transcript. This keeps archive generation and query completely untouched.

## Risk Assessment

- **Low risk**: The change is additive. If AI fails to generate options, we can fall back to the current textarea-only mode.
- **Prompt quality**: Options need to feel natural and culturally appropriate for elderly Chinese users. The prompt must be carefully crafted.
- **Token budget**: Adding options increases the response size slightly. With 4 questions × 4 options × ~20 chars each, this adds ~320 chars — well within moonshot-v1-8k limits.
